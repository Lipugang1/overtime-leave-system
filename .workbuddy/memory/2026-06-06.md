## 修复手写签名保存失败

### 根因
签名 API (`api/auth/signature/route.ts`) 使用 `coze-coding-dev-sdk` 的 `S3Storage` 上传图片到对象存储，
依赖环境变量 `COZE_BUCKET_ENDPOINT_URL` 和 `COZE_BUCKET_NAME`。本地环境未配置这些变量，导致
`storage.uploadFile()` 抛出异常，前端表现为"保存签名"按钮无反应。

### 修复方案
重写签名 API，添加本地文件系统回退模式：
- 当 `COZE_BUCKET_ENDPOINT_URL` 或 `COZE_BUCKET_NAME` 未配置时，自动退回到 `public/signatures/` 目录存储
- POST 上传：将签名图片写入 `public/signatures/{user_id}_{timestamp}.{ext}`
- GET 获取：返回 `/{file_path}`（public 目录下的文件可直接通过 URL 访问）
- DELETE 删除：同时删除本地文件和数据库记录
- 同时兼容 Coze S3 模式（生产环境）

### 验证
- 上传签名：`POST /api/auth/signature` → 200，`{"signature_key":"signatures/xxx.png"}`
- 获取签名：`GET /api/auth/signature` → 200，`{"signature_url":"/signatures/xxx.png","has_signature":true}`
- HTTP 访问签名图片：200 OK
- 删除签名：`DELETE /api/auth/signature` → 200

## 修复申请单导出（DOCX）失败

### 根因
加班/调休的 DOCX 申请单导出（`export-docx/route.ts`）在两个地方依赖 `coze-coding-dev-sdk` 的 S3Storage：
1. `getSignatureImage()` — 读取签名图片嵌入 DOCX
2. 最终导出 — 上传生成的 .docx 到 S3 并返回预签名下载 URL

本地环境未配置 `COZE_BUCKET_ENDPOINT_URL`，两处均抛异常 → 前端报 "Storage endpoint not configured"。

### 修复
两个文件（`overtime/export-docx/route.ts`、`leave/export-docx/route.ts`）统一修复：

**1. getSignatureImage() 本地回退**
- 有 Coze S3 配置 → 走 S3 `readFile`
- 无配置 → 从 `public/` 目录读取签名文件（按 `signature_key` 路径）

**2. 导出文件上传本地回退**
- 有 Coze S3 配置 → 上传到 S3 + 生成预签名 URL
- 无配置 → 写入 `public/exports/overtime/` 或 `public/exports/leave/`，返回 `/exports/...` 直接 URL

### 验证
- 加班申请单 DOCX 导出测试：200 OK
- 文件生成：`public/exports/overtime/加班登记单_系统管理员.docx`
- download_url 返回本地路径，浏览器可直接下载
